# DevConnect — تقرير مراجعة الجودة والأمان

**النطاق:** الـ backend بالكامل (Express + Prisma + Socket.io + Zod + JWT) + الجزء الخاص بعرض المحتوى في الـ client (اللي ليه علاقة بالـ XSS).
**الطريقة:** مراجعة ساكنة (static review) لكل route و middleware و schema و lib وطبقة الـ socket، بالإضافة لكتابة اختبارات فعلية قابلة للتشغيل. ما اتعدّلش أي كود في المشروع غير إضافة ملفات الاختبار.
**التشغيل الفعلي:** الاختبارات اتشغّلت على **Postgres حقيقي** (نسخة throwaway مؤقتة على البورت 5433 عبر `embedded-postgres`) — **الداتابيز الحقيقي بتاعك ما اتلمسش خالص**. كل الأدلة موجودة في `src/tests/security.spec.ts`.

## 2. تقارير الأخطاء

### BUG-01 — أي مستخدم يقدر يدّي لنفسه دور RECRUITER (وصول جماعي لبيانات شخصية) · **عالية**
- **المتأثر:** `src/schemas/auth.ts:14`، `src/routes/auth.ts:79`؛ والمستهلك المميّز `src/routes/talent.ts:10-11`، `src/routes/shortlist.ts:11`.
- **خطوات إعادة الإنتاج:** `POST /api/auth/register` بـ `{ ..., "role": "RECRUITER" }` → بعدها `GET /api/talent/search`.
- **المتوقع:** صلاحية الـ recruiter ما تتاخدش من إدخال المستخدم؛ البحث في المواهب محصور على recruiters موثّقين.
- **الفعلي:** الـ `registerSchema` بيقبل `role: z.enum(["DEVELOPER","RECRUITER"])` مباشرة من الـ body. الحساب فورًا بيعدّي من `requireRole("RECRUITER")` ويقدر يعدّد بيانات كل مطوّر: `displayName, location, yearsExperience, specialty, availability, skills` — بما فيهم اللي محددين `NOT_LOOKING`، من غير أي موافقة ومن غير إمكانية انسحاب.
- **الإصلاح المقترح:** شيل `role` من `registerSchema` (خلّي الافتراضي `DEVELOPER` للكل). فعّل حسابات الـ recruiter عبر مسار منفصل موثّق (دعوة/أدمن/فحص دومين). لو الفكرة إن التسجيل مفتوح فعلًا، على الأقل اعمل خيار opt-in لكل بروفايل ("ظاهر للـ recruiters") + توثيق هوية الـ recruiter.
- **الدليل الفعلي:** رجّع `candidates=4` لـ recruiter صنعناه بأنفسنا.
- **✅ اتصلّح (نهج الـ opt-in):** تسجيل الـ recruiter فضل مفتوح حسب اختيار المنتج، والضرر (تسريب البيانات) اتقفل بموافقة صريحة. أضفنا `Profile.discoverable Boolean @default(false)` (`prisma/schema.prisma`)؛ و`GET /api/talent/search` دلوقتي بيطبّق دايمًا فلتر `{ discoverable: true }` (`routes/talent.ts`)، فالمطورين اللي مش موافقين ما يظهروش خالص. عرّضنا الحقل في `updateProfileSchema` + `GET /api/profiles/me`، وضفنا toggle "Discoverable by recruiters" في صفحة تعديل البروفايل (`client/src/pages/EditProfile.tsx`). **التحقق:** سويت `security.spec.ts` قسم BUG-01 بقى 4/4 ناجح (غير الموافقين مش بيظهروا، الموافق بيظهر، وتفعيل واحد ما بيكشفش الباقي)؛ واتأكدنا من دورة الـ UI كاملة في المتصفح (الـ toggle بيتحفظ ويفضل متعلّم بعد الـ reload). **خطوة الإنتاج:** طبّق تغيير الـ schema على الداتابيز الحقيقي (`npx prisma db push` أو migration: `prisma migrate dev --name add_profile_discoverable`) قبل النشر.

### BUG-02 — الموارد الفرعية لبوست المجتمع الخاص بتتخطى فحص الرؤية (IDOR) · **عالية**
- **المتأثر:** `src/routes/posts.ts` — `GET /:id/comments` (323)، `GET /:id/reactions` (568)، `GET /:id/reposts` (295)، `POST /:id/like` (199)، `POST /:id/comments` (349)، `POST /:id/repost` (251).
- **خطوات إعادة الإنتاج:** الـ owner ينشر في مجتمع **خاص**. شخص مش عضو ومعاه الـ id بينادي `GET /api/posts/:id/comments` (200، بيسرّب المحتوى) و`POST /api/posts/:id/like` / `.../comments` (بينجحوا).
- **المتوقع:** نفس حماية الـ permalink `GET /api/posts/:id` اللي بيطبّق `communityVisibility()` (`posts.ts:187`) وبيرجّع 404 لغير الأعضاء.
- **الفعلي:** الـ handlers الفرعية بتفحص `assertNotBlocked` بس (أو ولا حاجة). العضوية/الرؤية ما بتتفحصش أبدًا، فبتتسرّب نقاشات المجتمع الخاص وقوائم المتفاعلين، والغريب يقدر يحقن لايكات/كومنتات (وكمان يولّد إشعارات لصاحب البوست).
- **ليه مهم:** الـ cuid مش سرّي — بيظهر في روابط الإشعارات وروابط المشاركة، والأعضاء القدامى محتفظين بيه. ده بيلغي الغرض الأساسي من "المجتمع الخاص".
- **الإصلاح المقترح:** ضيف guard مشترك: هات `communityId` بتاع البوست، وطبّق `communityVisibility(viewerId)` (للقراءة) أو فحص عضوية صريح (للكتابة) على كل مورد فرعي، وارجّع 404/403 لغير الأعضاء — زي الـ permalink بالظبط.
- **الدليل الفعلي:** الغريب أخد 200/201 بينما الـ permalink رجّع 404 صح.
- **✅ اتصلّح:** أضفنا `assertPostVisible(postId, viewerId)` (`routes/posts.ts`) بيعيد استخدام `communityVisibility()` وبيرمي 404 لو البوست مش مرئي. الـ 3 مسارات GET (`/comments`, `/reactions`, `/reposts`) بينادوها في الأول؛ ومسارات الكتابة (`/like`, `/comments`, `/repost`) غيّرنا فيها `findUnique` لـ `findFirst({ where: { id, ...communityVisibility(viewerId) } })`، فغير الأعضاء بياخدوا 404. **التحقق:** قسم BUG-02 في `security.spec.ts` بقى 8/8 ناجح — الغريب متحجوب على الست endpoints، والعضو لسه بيقرا ويعمل لايك عادي؛ و`communities.spec` (لايك على بوست عام) لسه أخضر.

### BUG-03 — قائمة أعضاء المجتمع الخاص وتفاصيله مكشوفة لغير الأعضاء · **متوسطة**
- **المتأثر:** `src/routes/communities.ts:334` (`GET /:slug/members` من غير أي فحص عضوية) و`:87` (`GET /:slug` بيرجّع `memberCount` + `memberPreview` بغضّ النظر عن الخصوصية).
- **خطوات إعادة الإنتاج:** غير العضو ينادي `GET /api/communities/<private-slug>/members` → قائمة كاملة بـ username/displayName/avatar/role.
- **المتوقع:** للمجتمعات `isPrivate` القائمة والـ preview للأعضاء بس (زي `/:slug/posts` اللي بيفحص فعلًا).
- **الفعلي:** مفيش فحص خصوصية على المسارين دول.
- **الإصلاح المقترح:** في الـ handlers الاتنين، لو `community.isPrivate` والمنادي مالوش صف `CommunityMember`، ارجّع الأعداد بس أو 403 — بنفس منطق مسار البوستات.
- **الدليل الفعلي:** رجّعت قائمة الأعضاء كاملة لشخص من بره.
- **✅ اتصلّح:** `GET /:slug/members` دلوقتي بيرجّع 403 لغير أعضاء الكوميونتي الخاص؛ و`GET /:slug` بيرجّع `memberPreview` فاضية لغير الأعضاء (العدد بس بيفضل — مش بيسرّب هوية حد). الأعضاء مش متأثرين. **التحقق:** قسم BUG-03 في السويت 3/3 ناجح (غير العضو 403 + preview فاضية، والعضو لسه بيشوف الروستر).

### BUG-04 — الحظر بيمنع التفاعل بس مش القراءة · **متوسطة**
- **المتأثر:** مسارات القراءة اللي مفيهاش فحص حظر/رؤية: `src/routes/profiles.ts:106` (`GET /:username`)، `src/routes/posts.ts:394` (`GET /user/:username`)، `src/routes/search.ts`، `profiles.ts:148` (github-projects).
- **خطوات إعادة الإنتاج:** A بيحظر B (`POST /api/moderation/block/A`). B لسه بياخد 200 على `GET /api/profiles/A` و`GET /api/posts/user/A` وبيلاقي A في `/api/search`.
- **المتوقع:** حسب تعليق الـ schema ("لو A حظر B، مايشوفش بعض")، الحظر المفروض يخفي البروفايلات/البوستات/البحث بين الطرفين.
- **الفعلي:** `assertNotBlocked` مطبّقة على الكتابة (صداقة/متابعة/رسائل/لايك/كومنت/ريبوست) بس مش على القراءة، فالمحظور لسه يقدر يراقب اللي حظره.
- **الإصلاح المقترح:** ضيف فحص حظر على قراءة البروفايل/بوستات-المستخدم/البحث (استبعد أو 404 لو في حظر في أي اتجاه). اعملها دالة مركزية زي `assertNotBlockedOrHide(viewer, target)`.
- **الدليل الفعلي:** 200 على البروفايل والبوستات بعد الحظر.
- **✅ اتصلّح:** أضفنا `isBlockedBetween(a, b)` (`lib/blocks.ts`). `GET /api/profiles/:username` و`/github-projects` و`GET /api/posts/user/:username` بيرجّعوا 404 لو في حظر في أي اتجاه (إخفاء متبادل من غير ما نكشف الحظر)؛ و`GET /api/search` بيستبعد المحظورين وبوستاتهم عبر شرط `notBlocked`. **التحقق:** قسم BUG-04 في السويت ناجح (المحظور بياخد 404 على البروفايل/البوستات ومش بيلاقيه في البحث؛ وطرف تالت مش محظور لسه بيشوف البروفايل).

### BUG-05 — إعادة تعيين الباسورد ما بتبطّلش الجلسات القايمة · **متوسطة**
- **المتأثر:** `src/lib/jwt.ts` (توكنات stateless مدتها 7 أيام، من غير version/jti)، `src/routes/auth.ts` — `reset-password` (308) و`login`.
- **خطوات إعادة الإنتاج:** مهاجم سرق JWT الضحية. الضحية يعمل flow نسيان/إعادة تعيين كامل. التوكن المسروق يفضل شغّال لحد ما ينتهي بعد 7 أيام.
- **المتوقع:** إعادة تعيين الباسورد (إجراء التعافي القياسي من الاختراق) تبطّل كل التوكنات القايمة.
- **الفعلي:** مفيش آلية إبطال؛ الـ JWT بيفضل صالح بغضّ النظر عن تغيير الباسورد.
- **الإصلاح المقترح:** ضيف عمود `tokenVersion` (أو `sessionsValidFrom`) على `User`؛ حطّه في الـ JWT وقارنه في `requireAuth`/socket auth؛ زوّده عند `reset-password` (وكمان لخيار "تسجيل خروج باقي الأجهزة").

### BUG-06 — تخطّي الـ rate limit عن طريق تزوير IP · **متوسطة (يعتمد على النشر)**
- **المتأثر:** `src/index.ts:29` (`app.set("trust proxy", 1)`) + `src/middleware/rateLimit.ts` (limiters بالـ IP).
- **خطوات إعادة الإنتاج:** لو التطبيق متاح من غير proxy موثوق واحد بالظبط قدّامه، المهاجم يبعت `X-Forwarded-For` متغيّر؛ Express بيعتبر كل قيمة IP جديد، فيتخطّى `authLimiter` (10/15د) و`apiLimiter` (300/15د) → brute-force / إغراق.
- **المتوقع:** الـ rate limiting مبني على IP العميل الحقيقي.
- **الفعلي:** `trust proxy: 1` بيثق عمياني في أول hop؛ صح بس لما يكون في proxy حقيقي دايمًا بيحطّه.
- **الإصلاح المقترح:** اضبط `trust proxy` على البنية الفعلية (عدد/CIDR الـ proxy المعروف، أو `false` لو مكشوف مباشرة). فكّر في `keyGenerator` ما يعتمدش على هيدرز قابلة للتزوير، وضيف throttling على مستوى الحساب في الـ login.

### BUG-07 — استنتاج المستخدمين عبر زمن نسيان الباسورد · **منخفضة**
- **المتأثر:** `src/routes/auth.ts:271`.
- **التفصيل:** الرد (شكل وحالة) متطابق للإيميل الموجود وغير الموجود صح، بس فرع "الإيميل موجود" بيعمل كتابة DB + إرسال SMTP، فبيعمل فرق زمني ملحوظ.
- **الإصلاح المقترح:** وحّد الزمن (اعمل الشغل async بعد الرد، أو ضيف حدّ زمني ثابت).

### BUG-08 — عنوان إيميل الرسالة مش مُنظّف (escaped) · **منخفضة**
- **المتأثر:** `src/socket.ts:211` — العنوان بيستخدم `${senderName}` خام بينما الجسم بيستخدم `safeSenderName` صح.
- **التفصيل:** الـ `displayName` (≤60 حرف، من غير منع أسطر جديدة) بيدخل العنوان من غير escaping. الـ HTML مش بيتفسّر في العنوان و nodemailer بيرفض أسطر الهيدر الجديدة، فالتأثير محدود — بس دفاع غير متّسق.
- **الإصلاح المقترح:** استخدم `safeSenderName` في العنوان كمان، وشيل أحرف التحكّم.

### BUG-09 — الفيد بيحمّل لحد 600 صف في الميموري كل طلب · **منخفضة**
- **المتأثر:** `src/routes/posts.ts:75-151` (`FETCH_CAP = 300` بوست + 300 ريبوست، دمج/ترتيب في JS، تقطيع بالـ offset).
- **التفصيل:** تبسيط مقصود. تمام دلوقتي، بس التكلفة بتكبر مع البيانات، والـ offset paging ممكن يكرّر/يسقط عناصر لما بوستات جديدة تيجي.
- **الإصلاح المقترح:** انقل لـ keyset pagination على query موحّد لما البيانات تكبر.

**نتائج إيجابية (دفاعات شغّالة صح):** bcrypt بتكلفة 12؛ توكن إعادة التعيين متخزّن كـ SHA-256 hash بصلاحية 30 دقيقة واستخدام واحد؛ رسائل auth عامة (مفيش استنتاج مستخدمين من الـ login)؛ كوكي `state` ضد الـ CSRF بمقارنة `timingSafeEqual`؛ إيميل GitHub العام مش بيتربط تلقائي (حماية من سرقة الحساب)؛ `httpUrl()` بيمنع روابط `javascript:`/`data:` على كل روابط المستخدم؛ `react-markdown` مستخدم من غير `rehype-raw` و`highlight.js` بيعمل escape (مفيش stored XSS عبر البوستات/السنيبتات)؛ Prisma parameterization (مفيش SQLi)؛ الـ `passwordHash` عمره ما بيتحدّد في أي رد؛ auth للـ socket + فحص عضوية المحادثة على `message:send` و`typing`؛ helmet مفعّل؛ حجم الـ JSON محدود بـ 1 ميجا.

- **النتيجة الفعلية على الكود الحالي:** `11 passed · 10 vulnerabilities · 0 test-failures`. وكل الـ specs القديمة (`auth` / `chat` / `communities` / `talent`) بتنجح على DB نظيف.
- **القضايا المؤكدة:** 2 عالية، 4 متوسطة، 3 منخفضة. **BUG-01 → BUG-04 اتصلّحوا دلوقتي** (متحقّق منهم: السويت بترجّع `29 passed · 0 vulnerabilities · 0 test-failures`، والـ specs القديمة لسه خضرا)؛ BUG-05 → BUG-09 لسه مفتوحين (إبطال الجلسات، rate-limit بالـ IP، تسريب زمني، escaping عنوان الإيميل، paging الفيد).

**أهم المخاطر بالترتيب:**
1. **BUG-01** — التسجيل الذاتي كـ RECRUITER بيحوّل بيانات المطورين لـ endpoint مفتوح عمليًا.
2. **BUG-02** — المجتمعات الخاصة مش خاصة فعلًا بالنسبة للتفاعل مع البوستات وموارده الفرعية.
3. **BUG-03 / BUG-04** — قوائم الأعضاء الخاصة وقراءات المستخدم المحظور بتسرّب معلومات المنتج بيوحي إنها محميّة.
4. **BUG-05 / BUG-06** — فجوة إبطال الجلسات وتخطّي الـ rate limit بتزوير IP بيضعّفوا التعافي من الاختراق ودفاعات الـ brute-force.
